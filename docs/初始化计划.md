### 阶段 C：编排主链路收口（P1）

需要完成：

1. 为 `html_worker` / `manim_worker` / `svg_worker` 定义统一执行适配接口。
2. 让 worker 执行结果正式回写：
   - 长期上下文

   - `events.jsonl`
3. 为 reviewer 明确结构化输入，而不是只依赖任务状态。
4. 补齐 HTML / Manim / SVG 产物 schema 与文件组织约定。

验收标准：

- 至少有一个真实 worker 不依赖手工模拟就能走完：
  - `dispatch.context_pack`
  - `worker.progress`
  - `worker.result`
  - `leader.commit`

### 阶段 D：人工审核闭环（P1）

目标：把 `review.outputs` 从“结构化占位关卡”收口成“人工确认后才可放行”的真实审核流程。

需要完成：

1. 明确 `reviewer` 只负责生成审核草案，最终放行必须由人工确认。
2. 人工审核需要至少支持两种动作：
   - `approve`
   - `return`
3. 人工 `return` 时，必须同时固化两类信息：
   - 上下文植入位置：写入会话级返工记录，供下一轮 `context-pack` 读取
   - prompt 植入位置：写入独立的返工提示段，避免把人工意见埋在普通 summary 里
4. 正式 `review.report` 只记录人工确认后的审核结论；AI reviewer 草案不直接视为通过。

验收标准：

- `review.outputs` 未经人工确认不得进入 `post_produce.outputs`
- 人工打回后，下一轮目标角色的 context packet 中能看到独立的“人工打回意见”段

### 阶段 E：控制台接线（P1）

目标：把已有控制台骨架从 mock 数据切换为真实编排状态视图，并承载人工审核动作。

需要完成：

1. 前端接入 `GET /runtime`、`POST /tasks/update`、`POST /context-pack`
2. 展示真实阶段流、任务看板、上下文包、事件流
3. 增加人工审核面板：
   - 审核草案预览
   - 人工 `approve`
   - 人工 `return`
   - 打回意见编辑与目标角色选择
4. 保持前端只通过 `backend/` 访问状态，不直接读 `runtime/`

验收标准：

- 能在控制台里查看 `stage1-golden-path` 或 `html-worker-validation` 的真实 runtime 数据
- 能从控制台触发一次人工审核通过或打回，并看到对应 runtime 回写结果

## 五、第三方资源同步的当前建议

当前不建议在未确认上游源码路径前直接重跑：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\sync-thirdparty-assets.ps1
```

原因：

- 该脚本会先清空目标目录再复制白名单资源。
- 如果默认探测的上游目录不存在，就可能把当前仓库内已经存在的第三方资产清掉。

正确做法：

1. 先确认上游源码路径存在。
2. 必要时显式传入：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\sync-thirdparty-assets.ps1 `
  -HtmlSkillSource "<AI-Animation-Skill-main 路径>" `
  -HyperframesSource "<hyperframes-main 路径>" `
  -ManimSkillSource "<manim skill 路径>"
```

3. 同步后重新检查 `runtime/thirdparty-sync-report.json` 是否已经写成当前工作区路径。

## 六、结论

当前 ManiMind 的初始化重点已经明确为两件事：

1. 把环境基线收口，让 `doctor` 和测试基线稳定可复现。
2. 在现有 runtime、CLI、API 基础上，把真实 worker 执行链和“人工确认型 reviewer”闭环做实。

在这两件事完成前，不建议把注意力转去做更多新生成器或大规模后处理集成。
