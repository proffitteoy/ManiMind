# ManiMind 状态闭环与控制台路线

本文件记录当前仓库的阶段性判断，以及下一步应优先落地的产品与技术路线。

## 一、当前判断

### 1. 编排内核已经成立

当前仓库已经形成可运行的编排内核：

- `models.py` 定义语义骨架。
- `workflow.py` 负责把 manifest 编译成上下文蓝图、角色分布和任务 DAG。
- `task_board.py` 负责 owner / blocker / review gate。
- `runtime_store.py` 负责快照落盘和事件日志。

这说明 ManiMind 已经过了“搭空架子”阶段。

### 2. P0 主矛盾是状态闭环

当前最需要补的不是内容生成，而是把 runtime state 做成唯一事实源。

现在 `task-update` 只从 `execution-tasks.json` 回填任务状态，随后也只更新 `execution-tasks.json`，不会同步更新 `state.json` 和 `project-plan.json` 里的 `current_stage`。长期运行后会出现：

```text
execution-tasks.json: 某些任务已经 completed
state.json: current_stage 仍然是 prestart
project-plan.json: 仍然是旧计划快照
```

这会导致“任务机”和“阶段机”分叉。

### 3. 上下文装配仍是 metadata packet

`build_context_packet(...)` 当前主要输出：

- `key`
- `scope`
- `summary`
- `writer_role`
- `consumer_roles`
- `lifecycle`
- `invalidation_rule`

它还没有读取真实上下文正文，因此目前更接近 prompt scaffold，而不是 assembled prompt。

### 4. 当前还存在两个治理缺口

- `consumer_roles` 已经建模，但还没有被严格用于过滤上下文。
- `stage_allowed` 目前只是提示字段，还不是 CLI / API 的硬约束。

## 二、明确优先级

| 优先级 | 模块 | 动作 |
|:---|:---|:---|
| P0 | `runtime_store.py` | 实现 runtime 统一加载与阶段同步 |
| P0 | `task-update` | 每次任务推进后同步更新 `current_stage` |
| P0 | `context_assembly.py` | 按 `consumer_roles` 过滤上下文 |
| P1 | `context-pack` | 对非法 `stage_allowed=false` 默认阻断 |
| P1 | `runtime_store.py` | 写入改成 atomic write，避免半写文件 |
| P2 | `PromptSectionCache` | 之后再考虑持久化 cache registry |
| P2 | `review_checkpoints` | 之后再做自动执行器 |

一句话判断：

```text
下一步不是加功能，而是把 runtime state 做成唯一事实源。
```

## 三、推荐改造顺序

```text
第一步：实现 ProjectRuntime 统一加载器
第二步：实现 current_stage 从 execution_tasks 派生
第三步：task-update 后同步写 state.json + execution-tasks.json + project-plan.json
第四步：把 context-pack 改成先 load runtime，再 build packet
第五步：再做真实上下文内容装配
```

推荐新增运行时加载结构：

```python
@dataclass(slots=True)
class ProjectRuntime:
    project_id: str
    state: dict[str, Any]
    context_records: list[dict[str, Any]]
    execution_tasks: list[dict[str, Any]]
    project_plan: dict[str, Any] | None
```

并实现：

```python
def load_project_runtime(plan: ProjectPlan) -> ProjectRuntime:
    ...
```

## 四、阶段派生建议

建议把 `current_stage` 变成从任务状态派生的结果，而不是独立漂浮字段。

```python
def derive_current_stage(plan: ProjectPlan) -> PipelineStage:
    status = {task.id: task.status for task in plan.execution_tasks}

    if status.get("post_produce.package") == TaskStatus.COMPLETED:
        return PipelineStage.DONE

    if status.get("post_produce.package") == TaskStatus.IN_PROGRESS:
        return PipelineStage.POST_PRODUCE

    if status.get("review.outputs") == TaskStatus.COMPLETED:
        return PipelineStage.PACKAGE

    if status.get("review.outputs") == TaskStatus.IN_PROGRESS:
        return PipelineStage.REVIEW

    render_tasks = [
        task for task in plan.execution_tasks
        if task.id.startswith("render.")
    ]
    if render_tasks and all(task.status == TaskStatus.COMPLETED for task in render_tasks):
        return PipelineStage.REVIEW

    if status.get("plan.storyboard") == TaskStatus.COMPLETED:
        return PipelineStage.DISPATCH

    if status.get("summarize.research") == TaskStatus.COMPLETED:
        return PipelineStage.PLAN

    if status.get("ingest.sources") == TaskStatus.COMPLETED:
        return PipelineStage.SUMMARIZE

    if status.get("ingest.sources") == TaskStatus.IN_PROGRESS:
        return PipelineStage.INGEST

    return PipelineStage.PRESTART
```

在 `persist_task_update(...)` 里建议同步：

```python
plan.current_stage = derive_current_stage(plan)
```

并额外写回：

```text
runtime/projects/<project_id>/state.json
runtime/projects/<project_id>/execution-tasks.json
runtime/projects/<project_id>/project-plan.json
```

## 五、前端产品定位

ManiMind 的前端不应该先做成“聊天窗口 + 文件上传”的普通 AI 工具，而应该呈现为一个项目编排控制台。

核心目标是让用户清楚看到：

```text
输入了什么
→ 系统理解成什么项目
→ 当前在哪个阶段
→ 哪些 Agent 在做什么
→ 产物生成到哪里
→ 审核是否通过
```

推荐主界面结构：

```text
左侧：项目与资源
中间：流程编排 / 任务看板
右侧：当前上下文 / Agent 输出 / 审核意见
底部或独立页：产物预览与导出
```

### 首页

首页只解决一件事：把输入收集完整，然后生成 manifest。

```text
新建 ManiMind 项目
├─ 上传论文 PDF
├─ 上传/粘贴用户笔记
├─ 选择目标受众
├─ 选择输出风格
└─ 创建项目
```

### 项目页

项目页应做成“编排驾驶舱”。

左栏负责显示：

```text
项目基本资料
输入资源
长期上下文
```

中间主区优先显示：

```text
阶段流 + 任务链 / 任务 DAG
```

右栏负责显示当前任务：

```text
当前任务
执行角色
读取上下文
写入目标
Prompt Sections
运行与状态操作
```

### MVP 前端

第一版只需要证明编排内核价值：

```text
1. 上传 PDF / 笔记 / manifest
2. 生成 ProjectPlan
3. 展示阶段流
4. 展示 ExecutionTask 列表
5. 点击任务生成 context packet
6. 手动更新任务状态
7. 展示 runtime/projects/<project_id>/ 下的状态文件
```

## 六、技术架构建议

推荐总体结构：

```text
前端控制台
+ 后端编排 API
+ 文件型 runtime
+ Worker 执行器
```

更完整的分层如下：

```text
┌─────────────────────────────────────────────────────────────┐
│                         前端层                               │
│  Next.js / React / Tailwind / shadcn/ui / React Flow        │
└───────────────────────────────┬─────────────────────────────┘
                                │ HTTP / SSE
┌───────────────────────────────▼─────────────────────────────┐
│                         API 层                               │
│  FastAPI                                                     │
└───────────────────────────────┬─────────────────────────────┘
                                │ 调用编排内核
┌───────────────────────────────▼─────────────────────────────┐
│                     ManiMind 编排核心层                      │
│  models.py / workflow.py / context_assembly.py /            │
│  task_board.py / runtime_store.py                           │
└───────────────────────────────┬─────────────────────────────┘
                                │ 读写
┌───────────────────────────────▼─────────────────────────────┐
│                      Runtime 状态层                          │
│  runtime/projects/<project_id>/ ...                         │
└───────────────────────────────┬─────────────────────────────┘
                                │ 分发任务
┌───────────────────────────────▼─────────────────────────────┐
│                         Worker 层                            │
│  explorer / planner / coordinator / html / manim / svg /    │
│  reviewer                                                    │
└─────────────────────────────────────────────────────────────┘
```

### 第一版技术选择

- 前端：`Next.js + React + Tailwind + shadcn/ui + React Flow`
- 后端：`FastAPI + Pydantic`
- Runtime：`JSON 文件 + atomic write + file lock`
- 通信：`REST API`，必要时加 `SSE`
- Worker：第一版同步执行，后面再队列化

### 第一版不建议做的事

- 不要先做微服务。
- 不要让前端直接读写 runtime 文件。
- 不要先把系统做成云端 AI 视频平台。
- 不要先做成视频编辑器形态。

## 七、API 建议

第一版建议只做这些 API：

```text
POST   /api/projects
GET    /api/projects
GET    /api/projects/{project_id}

POST   /api/projects/{project_id}/plan
GET    /api/projects/{project_id}/runtime

GET    /api/projects/{project_id}/tasks
POST   /api/projects/{project_id}/tasks/{task_id}/update

POST   /api/projects/{project_id}/context-pack
GET    /api/projects/{project_id}/contexts

GET    /api/projects/{project_id}/outputs
GET    /api/projects/{project_id}/outputs/{output_id}

POST   /api/projects/{project_id}/workers/{agent_mode}/run
GET    /api/projects/{project_id}/events
```

其中关键接口包括：

- `POST /plan`：根据 manifest 生成 `ProjectPlan` 并写入 runtime。
- `GET /runtime`：返回当前完整 runtime 状态。
- `POST /tasks/{task_id}/update`：更新任务状态，并同步刷新 `current_stage`。
- `POST /context-pack`：给指定 Agent 和任务生成上下文包。

## 八、Runtime 与执行器建议

推荐 runtime 目录继续维持文件型设计：

```text
runtime/
└─ projects/
   └─ <project_id>/
      ├─ state.json
      ├─ project-plan.json
      ├─ context-records.json
      ├─ execution-tasks.json
      ├─ context/
      ├─ outputs/
      ├─ snapshots/
      └─ events.jsonl
```

其中 `events.jsonl` 应作为关键事件总线，记录：

- `project.created`
- `task.updated`
- `stage.changed`

Worker 第一版建议采用统一接口的同步执行器，而不是并发 Agent 集群：

```python
class BaseWorker:
    agent_mode: AgentMode

    def run(self, runtime: ProjectRuntime, task_id: str) -> WorkerResult:
        ...
```

## 九、阶段化路线

### 阶段 A：控制台 MVP

把现有 `plan/context-pack/task-update` 包装成 Web UI。

### 阶段 B：真实内容接入

把 context packet 推进成 assembled prompt。

### 阶段 C：Worker 执行器

支持任务点击后真正生成内容，并记录 prompt / output 快照。

### 阶段 D：产物预览与审核闭环

重点展示脚本、分镜、Manim、HTML、审核报告。

### 阶段 E：异步任务与多项目

在状态闭环稳定后，再考虑 SSE、后台任务、SQLite/PostgreSQL、任务队列。

## 十、最终判断

ManiMind 当前最该构建的是：

```text
可视化编排控制台
+ runtime 唯一事实源
+ 可插拔 Worker 执行器
+ 可审计 Prompt / Output 快照
```

在这四件事打稳之前，不建议把重点放到 PDF 解析、LLM 内容生成器或视频编辑器式前端上。
