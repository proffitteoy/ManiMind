# ManiMind 审核与代码审计标准

本文件对应审核 Agent 与人工代码审阅规则，基于当前已实现的编排层能力。

## 一、审核对象

- 角色模式与职责边界是否一致
- 上下文写入是否符合 writer/consumer/lifecycle/invalidation 约束
- 任务状态推进是否遵守 owner/blocker/verification 规则
- 审核关卡是否被绕过
- 第三方能力接入是否保持“编排层调用”而非“内部重写”

## 二、P0 关卡

### P0-1 数学正确性

- 公式、定理、推导关系不能与论文/笔记冲突
- 科普转述必须可追溯到结构化上下文

### P0-2 审核强制性

- `review.outputs` 未完成时，不得推进 `post_produce.outputs`
- `post_produce.outputs` 未完成时，不得推进 `package.delivery`
- 任何“直接进入后处理”的路径视为阻塞

### P0-3 状态机正确性

- blocker 未完成时，任务不得进入 `in_progress` 或 `completed`
- 非 owner 角色（且非 `lead`）不得修改任务状态

### P0-4 Runtime 单一事实源

- `task-update` 后必须同步刷新 `current_stage`
- `state.json`、`execution-tasks.json`、`project-plan.json` 不得长期相互矛盾
- 禁止只更新任务快照而不更新阶段状态

## 三、P1 关卡

### P1-1 上下文治理完整性

- 每个核心上下文都要有 `writer_role` 与 `consumer_roles`
- `lifecycle` 和 `invalidation_rule` 不得缺失
- `consumer_roles` 不能只存在于字段层，必须实际参与 `context-pack` 过滤

### P1-2 角色模式一致性

- `read_only` 角色不得声明写入目标
- `verify_only` 角色不得承担内容生产任务
- `structured_write` 角色写入范围必须与 `owned_outputs` 一致
- 非法阶段请求不应默认继续执行，除非显式允许

### P1-3 可执行性与可追溯性

- `context-pack` 输出应包含阶段可执行判断（`stage_allowed`）
- `task-update` 输出应包含 mutation 结果与原因字段

## 四、P2 关卡

- 风格一致性与叙事节奏一致性
- 输出清单命名规范与路径规范一致性
- 提示词分段缓存命中策略是否合理

## 五、代码审计清单

1. 是否在 `src/manimind/` 落地，而非复制第三方内部实现。
2. 是否混写长期/短期上下文路径。
3. 是否出现无生命周期定义的新状态字段。
4. 是否存在不可追溯的任务依赖或隐式状态迁移。
5. 是否存在审核失败仍继续执行的代码路径。

## 六、建议输出格式

- 结论：通过 / 阻塞 / 需修改
- 严重问题列表（按 P0/P1/P2）
- 受影响模块
- 建议修复动作
- 是否允许进入后处理
