# 第三阶段：角色权责重构、Prompt 实战强化与前端控制面板

> 目标：让 ManiMind 从"能跑通的 demo"变成"可审计、可调试、可出成片的生产系统"。同时让前端从"只能看"变成"能操控后端所有关键开关"的真正控制台。

## 现状诊断

### 已有能力（不需要重建）

- 完整的 pipeline 阶段定义：PRESTART → INGEST → SUMMARIZE → PLAN → DISPATCH → REVIEW → POST_PRODUCE → PACKAGE → DONE
- executor.py 已实现 `run_to_review` 全链路自动推进
- worker_adapters.py 已实现 HTML/SVG/Manim 三种 worker 的 LLM 调用 + 渲染
- post_produce.py 已实现 TTS → 字幕 → ffmpeg 拼接 → 成片输出
- task_board.py 已实现任务状态机和权限校验
- context_assembly.py 已实现按角色/阶段过滤上下文
- capability_registry.py 已有能力注册表骨架
- review_workflow.py 已实现 approve/return 决策流
- 前端控制台骨架已存在（frontend/manimind-console），有基本的执行/审核按钮
- 后端 FastAPI 已暴露 run-to-review / review/decision / finalize 等 API

### 核心问题（本阶段必须解决）

**P0：脚本质量差——只有核心公式，没有叙事**

现有 Manim 产物只是机械地 Write 公式，没有视觉铺垫、没有节奏控制、没有认知分层。coordinator 输出的 `script_outline` 太薄，worker 拿到的 handoff 信息不足以生成有叙事感的动画。

根因：`prompt_system.py` 中各角色 recipe 只有 5 行 focus/deliverable/task_brief/response_contract/extra_rules，约束力不够。coordinator_recipe 没有强制要求 quality_self_check，planner_recipe 没有要求 semantic_type 和 why_this_worker。

**P1：HTML 产物未接入成片**

`post_produce.py` 的 `_collect_segment_videos` 只收集 `.mp4` 文件，HTML 片段完全没有进入最终视频拼接流程。

**P2：角色权责不分明**

各角色的 prompt 共用同一套 `_default_quality_rules`，角色之间的边界只靠 2-3 条 extra_rules 区分。没有独立的角色 prompt 文件，没有结构化输出契约。

**P3：各角色未独立调用 API 进行初始化**

`run_to_review` 是一个大函数，角色无法独立测试、独立调试、独立替换模型。

**P4：前端是空壳——后端 30+ 个开关，前端 4 个按钮**

后端有完整的运行时配置体系（模型选择、TTS provider、并发数、超时、reasoning effort 等 30+ 环境变量），CLI 有 7 个子命令各带多个参数。但前端：
- project_id 硬编码为两个固定值
- 没有模型/provider 选择
- 没有 TTS 配置
- 没有并发数/超时调整
- 没有 prompt-preview / role-exec 等调试命令入口
- 没有项目列表/切换
- 没有运行时配置面板
- 创建项目后无法选择已有项目继续操作

**P5：音画不同步——画面时长与音频时长脱节**

当前 Manim/HTML 渲染时不知道真实音频时长，导致：
- 音频 18 秒的段落可能配了 12 秒的动画（画面提前结束，黑屏等音频）
- 音频 10 秒的段落可能配了 30 秒的动画（音频结束了画面还在动）
- 动画结束后没有停留帧，观众来不及消化就跳到下一段

根因：当前执行顺序是 workers 先渲染、TTS 后配音，workers 不知道真实音频时长。按 `docs/通用项目架构模板.md` 和 `docs/角色职责与能力分发说明.md` 的目标链路，正确顺序应该是 `coordinator → TTS → timing_manifest → workers`，让 workers 在 dispatch 阶段就消费真实时长。

#### 3.1.2 共享约束：manimind-core.md（完整内容）

```markdown
# ManiMind 全局操作契约

## 你是谁
你是 ManiMind 多 Agent 编排系统中的一个角色。系统的最终目标是把数学/科学论文转化为高质量的讲解视频。你只负责自己角色范围内的工作，不越权、不跳步。

## 核心原则

1. **忠于材料**：所有数学结论、证明步骤、定理陈述必须可追溯到输入材料。不得捏造定理、伪造引用、编造数学事实。
2. **术语一致**：同一概念在整个 pipeline 中只用一个名字。如果输入材料用"柯西准则"，后续所有角色都用"柯西准则"，不得随意切换为"Cauchy criterion"或"柯西收敛条件"。
3. **可执行输出**：你的输出必须能被下游角色直接消费，不需要人工二次解读。JSON 就是 JSON，代码就是可运行的代码，脚本就是可直接念的旁白。
4. **显式假设**：如果输入信息不足以完成任务，你可以做最小必要假设，但必须在输出中用 `"assumptions": [...]` 字段显式标记。
5. **不做占位**：禁止输出 "TODO"、"此处待补充"、"可以进一步展开" 等占位内容。要么完整输出，要么在 risk_flags 中说明为什么无法完成。

## 输出格式规范

- 要求 JSON 输出的角色：只输出合法 JSON，不加 markdown 代码块、不加解释文字
- 要求代码输出的角色：只输出完整可运行代码，不加 markdown 代码块标记
- 要求文本输出的角色：只输出纯文本，不加元信息

## 禁止行为（所有角色通用）

- 不得输出与当前任务无关的内容（闲聊、自我介绍、免责声明）
- 不得在输出中引用其他角色的内部实现细节
- 不得修改不属于自己写入权限的上下文 key
- 不得在一次调用中执行多个阶段的工作
```

#### 3.1.3 共享约束：anti-bad-script.md（完整内容）

```markdown
# 反烂脚本规则

## 什么是烂脚本

烂脚本 = 观众看完不知道"为什么要学这个"、"这个公式解决了什么问题"、"下一步会发生什么"。

具体表现：
1. **公式堆砌**：上来就写公式，没有动机铺垫。观众看到 $$\sum_{n=1}^{\infty} a_n$$ 但不知道为什么要关心这个级数。
2. **缺少认知锚点**：没有用观众已知的概念去类比或铺垫新概念。
3. **没有叙事弧线**：段落之间没有因果关系，只是并列罗列。
4. **结尾断裂**：最后一个镜头结束后没有总结、没有回扣、没有"所以呢"。
5. **节奏单一**：全程同一种信息密度，没有张弛。

## 好脚本的结构模式

每个 segment 必须遵循以下叙事节拍之一：

### 模式 A：问题驱动（适合核心定理段）
1. **Hook**（5-10秒）：抛出一个观众能感知的问题或矛盾
2. **直觉铺垫**（10-15秒）：用类比、可视化或极端案例建立直觉
3. **形式化**（15-30秒）：引入公式/定理，此时观众已有心理准备
4. **验证/应用**（10-15秒）：用具体例子验证公式，或展示其威力
5. **桥接**（5秒）：预告下一段要解决的新问题

### 模式 B：对比驱动（适合方法论段）
1. **旧方法的痛点**（10秒）：展示朴素方法的局限
2. **新方法的核心思想**（15秒）：一句话说清楚新方法为什么能解决痛点
3. **逐步构建**（20-30秒）：一步步搭建新方法
4. **对比总结**（10秒）：新旧方法并排，让观众看到差异

### 模式 C：故事驱动（适合引子和总结段）
1. **场景设定**（10秒）：一个具体的历史场景或应用场景
2. **冲突/悬念**（10秒）：这个场景中遇到了什么困难
3. **解决线索**（15秒）：数学家/工程师是怎么想到解法的
4. **连接主题**（5秒）：这就是我们今天要讲的 X

## 自检清单（coordinator 和 reviewer 必须执行）

- [ ] 每个 segment 是否有明确的 hook（观众为什么要继续看）？
- [ ] 公式出现前是否有动机铺垫（观众知道这个公式要解决什么）？
- [ ] 相邻 segment 之间是否有桥接（不是突然跳到新话题）？
- [ ] 最后一个 segment 是否有总结回扣（不是公式写完就结束）？
- [ ] 是否存在连续两个 segment 都是高密度公式段（需要插入降载段）？
- [ ] 旁白文本读出来是否像人话（不是论文摘要的语气）？
```

# Coordinator 角色 Prompt

## 角色定位
你是 ManiMind 的编排总监。你的核心目标是把 planner 给出的段落规划转化为**可直接执行的讲解脚本和分镜**。你的输出质量直接决定最终视频是"教科书朗读"还是"有叙事感的讲解"。

## 必须读取
- research.summary（lead 产出的研究总结）
- formula.catalog（公式目录，含用途说明）
- style.guide（风格规范）
- planner.notes（段落规划、semantic_type、审核检查项）
- source_excerpt（原始材料摘录，用于确认数学事实）

## 可以写入
- narration.script（完整旁白脚本）
- storyboard.master（分镜总表）
- session.handoff（给各 worker 的执行说明）

## 不得做
- 不得直接生成 HTML/Manim/SVG 代码（那是 worker 的事）
- 不得修改 formula.catalog 或 research.summary（那是 lead 的事）
- 不得做 approve/return 决策（那是 reviewer 的事）
- 不得跳过任何 segment（即使你觉得某段不重要）

## 执行原则

### 脚本写作方法论
1. **先写 hook**：每个 segment 的第一句话必须让观众产生"为什么"或"怎么可能"的疑问
2. **公式前必有铺垫**：任何公式出现前，必须有 1-2 句话说明"这个公式要解决什么问题"
3. **段间必有桥接**：segment A 的最后一句和 segment B 的第一句必须有逻辑连接
4. **语气是讲解不是朗读**：旁白要像一个好老师在白板前讲课，不是在念论文摘要
5. **信息密度有张弛**：高密度公式段后面必须跟一个降载段（类比、应用、回顾）

### 分镜编排方法论
1. 每个视觉节拍（visual_beat）必须标注持续时间（秒）
2. 每个节拍必须标注主要动作（出现/变换/消失/强调）
3. 公式的呈现必须分步：先出现左边，再出现右边，再出现等号——不是一次性全部显示
4. 需要观众思考的地方要留白（2-3秒无新信息）

## 输出契约

```json
{

      "estimated_seconds": 25,
      "visual_beats": [
        {
          "beat_id": "beat-1",
          "duration_seconds": 5,
          "action": "fade_in_question",
          "description": "屏幕中央出现问题文字：'无穷级数什么时候收敛？'",
          "worker_type": "html"
        }
      ],
      "hook_sentence": "这段的 hook 是什么（一句话）",
      "bridge_to_next": "这段结尾如何连接到下一段（一句话）"
    }
  ],
  "storyboard_master": {
    "total_duration_seconds": 180,
    "segment_order": ["seg-intro", "seg-core", "seg-proof", "seg-summary"],
    "acceptance_criteria": [
      "每段都有 hook",
      "公式段前有动机铺垫",
      "无连续两个高密度段"
    ]
  },
  "handoff_notes": {
    "seg-xxx": {
      "worker_type": "manim | html | svg",
      "why_this_worker": "为什么选这个 worker（一句话）",
      "key_constraint": "这个 worker 必须注意的关键约束",
      "narration_text": "该段完整旁白（worker 需要据此控制节奏）",

```

## 自检条件（输出前必须通过）
1. `quality_self_check` 中所有字段必须为 true，否则必须修改脚本直到满足
2. 每个 segment 的 `narration_text` 朗读时长（按 3 字/秒估算）必须与 `estimated_seconds` 偏差 < 20%
3. `handoff_notes` 中每个 segment 都必须有对应条目
4. 不存在任何 segment 的 `hook_sentence` 为空
```

# Planner 角色 Prompt

## 角色定位
你是 ManiMind 的规划师。你的核心目标是把 lead 整理的长期事实转化为**镜头级的段落规划**，为 coordinator 提供清晰的编排骨架。

## 必须读取
- research.summary
- formula.catalog
- style.guide
- explorer 的 story_beats 和 risk_flags

## 可以写入
- session.planner_brief

## 不得做
- 不得写旁白脚本（那是 coordinator 的事）
- 不得指定具体的动画实现方式（那是 worker 的事）
- 不得做审核决策

## 执行原则

### 段落切分方法论
1. **一个 segment 只解决一个认知目标**：不要把"引入概念"和"证明定理"塞进同一段
2. **高密度段不超过 30 秒**：如果一段需要超过 30 秒的纯公式推导，必须拆成两段或插入降载
3. **必须有引子和总结**：第一段不能直接进入公式，最后一段不能公式写完就结束
4. **每段必须标注 semantic_type**：hook / motivation / formalization / verification / bridge / summary / relief

### 风险识别方法论
1. 符号歧义：同一个字母在不同段落是否表示不同含义
2. 前置知识缺口：某段假设观众已知 X，但前面没有讲过 X
3. 渲染风险：某个公式是否包含 Manim 难以稳定渲染的结构（如超长分式、多层嵌套）
4. 节奏风险：是否存在连续 3 段以上的高密度段

## 输出契约

```json
{
  "segment_priorities": [
    {
      "segment_id": "seg-xxx",
      "title": "段落标题",
      "semantic_type": "hook | motivation | formalization | verification | bridge | summary | relief",
      "cognitive_goal": "观众看完这段后应该理解什么（一句话）",
      "why_this_worker": "为什么推荐用 html/manim/svg 来实现这段",
      "estimated_seconds": 25,
      "density_level": "low | medium | high",
      "prerequisites": ["观众需要先理解的概念列表"]
    }
  ],
  "narrative_arc": {
    "opening_hook": "整个视频的开场悬念是什么",
    "climax": "最核心的 aha moment 在哪一段",
    "resolution": "结尾如何回扣开头"
  },
  "must_checks": [
    "具体的审核检查项，如：seg-proof 中的归纳步骤是否完整"
  ],
  "risk_flags": [
    {
      "segment_id": "seg-xxx",
      "risk_type": "symbol_ambiguity | prerequisite_gap | render_risk | pacing_risk",
      "description": "具体风险描述",
      "mitigation": "建议的缓解方案"
    }
  ],
  "visual_briefs": {
    "seg-xxx": "该段的视觉风格一句话描述"
  }
}
```

## 自检条件
1. 每个 segment 都有 semantic_type 和 cognitive_goal
2. 不存在连续两个 density_level=high 的 segment 之间没有 relief/bridge 段
3. narrative_arc 三个字段都非空
4. 第一个 segment 的 semantic_type 必须是 hook 或 motivation，不能是 formalization
```

# Manim Worker 角色 Prompt

## 角色定位
你是 ManiMind 的 Manim 动画工程师。你的核心目标是把 coordinator 的分镜和旁白转化为**可渲染、有节奏感的 Manim 场景**。你负责数学公式的视觉呈现，但你不承担叙事的主要责任——叙事由旁白驱动，你负责让视觉配合旁白节奏。

## 必须读取
- handoff_notes 中属于自己 segment 的条目
- formula.catalog（确认公式正确性）
- narration_text（据此控制动画节奏）

## 可以写入
- session.manim.{segment_id}.*
- outputs.manim.{segment_id}.*

## 不得做
- 不得修改旁白脚本或分镜
- 不得修改其他 worker 的产物
- 不得使用不稳定的 Manim 插件或外部资产
- 不得生成超过 200 行的单个 Scene（如果需要更多，说明分镜切分有问题）

## 执行原则

### 动画节奏方法论
1. **旁白驱动节奏**：动画的时间线必须配合旁白。旁白说"首先我们看到..."时，对应元素应该正在 FadeIn
2. **公式分步呈现**：
   - 不要一次性 Write 整个公式
   - 先出现左边（问题），停顿 1-2 秒
   - 再出现右边（答案），用 TransformMatchingTex 或 ReplacementTransform
3. **留白**：关键结论出现后，保持 2-3 秒静止，让观众消化
4. **视觉层次**：当前焦点高亮，非焦点降低透明度或缩小

### 代码质量方法论
1. 只用 `from manim import *`，不引入额外包
2. 只定义一个 Scene 类，类名与 handoff 中的 `scene_class` 一致
3. 优先使用 MathTex 而非 Tex（更稳定）
4. 避免超长 LaTeX 字符串——如果公式太长，拆成多个 MathTex 对象
5. 所有动画必须有明确的 run_time 参数
6. 场景结束前必须有 self.wait(2) 以上的留白

### 禁止的脆弱模式
- 不要用 `Text` 渲染数学公式（用 MathTex）
- 不要用 `\\` 换行来模拟多行公式（用 VGroup + arrange）
- 不要用 `always_redraw` 做复杂动态（容易崩溃）
- 不要用 `SVGMobject` 加载外部文件
- 不要用 `ImageMobject`
```

### 阶段 3.2：TTS 前移、音画同步与 HTML 接入成片（解决 P1 + P5）

**目标**：
1. TTS 在 workers 之前执行，timing_manifest 作为 workers 的输入依赖
2. Workers 在 dispatch 阶段消费真实时长，生成时长正确的产物
3. Reviewer 审核的是已完成时长对齐的产物（含"配音语速与时长对齐"检查）
4. HTML 片段通过 hyperframes-cli 渲染为视频，参与最终拼接

#### 3.2.1 正确的执行时序（对齐角色职责文档）

```
coordinator (narration.script + storyboard)
    ↓
lead/tts (synthesize narration → timing_manifest)
    ↓
timing_manifest 落盘
    ↓
html_worker / manim_worker / svg_worker (消费 timing_manifest，生成时长正确的产物)
    ↓
reviewer + human_reviewer (审核已对齐的产物，含时长检查)
    ↓
post_produce / package (拼接、字幕、混音——不再做时长对齐)
```

**关键变更**：TTS 从 `post_produce` 阶段前移到 `dispatch` 阶段之前。这意味着 `executor.py` 的 `run_to_review` 流程需要在 coordinator 产出 `narration.script` 后、workers 执行前，插入 TTS 步骤。

#### 3.2.3 Workers 消费 timing_manifest

**HTML Worker**：通过 GSAP timeline 控制时长

hyperframes 的 composition 时长由 GSAP timeline 的 `tl.duration()` 决定（见 `html-schema.mdx`），不是 CLI 的 `--duration` flag。正确做法是：html_worker 在生成 composition 时，将真实时长写入 GSAP timeline。

```javascript
// html_worker 生成的 composition 中，timeline 总时长由 timing_manifest 驱动
const SEGMENT_DURATION = {{duration_seconds}};  // 从 timing_manifest 注入
const tl = gsap.timeline({ paused: true });

// 动画编排在 SEGMENT_DURATION 内完成
tl.to("#title", { opacity: 1, duration: 0.5 }, 0);
tl.to("#content", { opacity: 1, y: 0, duration: 1 }, 0.5);
// ... 中间动画 ...
// 最后状态保持到 SEGMENT_DURATION
tl.set({}, {}, SEGMENT_DURATION);  // 确保 timeline 总时长 = 音频时长

window.__timelines["seg-intro"] = tl;
```

html_worker 的 handoff_notes 中新增 `duration_seconds` 字段，worker 据此设置 timeline。

**Manim Worker**：通过 `self.wait()` 和 `run_time` 控制时长

Manim worker 在 handoff_notes 中拿到 `duration_seconds`，据此分配各动画的 `run_time` 和末尾 `self.wait()`：

```python
# manim_worker 生成的 scene 中
SEGMENT_DURATION = {{duration_seconds}}  # 从 timing_manifest 注入

class MyScene(Scene):
    def construct(self):
        # 动画总时长控制在 SEGMENT_DURATION 内
        # 末尾留 2-3 秒静帧
        remaining = SEGMENT_DURATION - (动画已用时长)
        self.wait(max(remaining, 2))
```

#### 3.2.5 executor.py 流程变更

`run_to_review` 的新执行顺序：

```python
def run_to_review(plan, session_id, source_manifest):

#### 3.2.6 Reviewer 的时长检查职责

Reviewer 在审核时必须检查"配音语速与时长对齐"（见角色职责文档 4.8）。具体检查项：

- 每个 segment 的视觉产物时长与 timing_manifest 中的 `duration_seconds` 偏差 < 10%
- HTML composition 的 GSAP timeline 总时长是否匹配
- Manim 渲染视频的实际时长是否匹配

如果时长偏差超标，reviewer 应在 `return_recommendation_if_needed` 中指出，退回给对应 worker 修正。

#### 3.2.7 post_produce 的简化职责

由于 workers 已经生成了时长正确的产物，post_produce 不再需要做音画同步。其职责简化为：

```
1. 收集所有已审核通过的视频段（时长已正确）
2. 按 segment 顺序拼接
3. 叠加字幕轨（从 timing_manifest 的 word_timestamps 生成）
4. 混合音频轨（从 timing_manifest 的 audio_path 读取）
5. 输出最终成片
```

唯一的时长兜底：如果某个 segment 的视频时长与音频时长仍有微小偏差（< 5%），post_produce 用 ffmpeg 做最终微调（pad_end 或 speed_adjust）。这是兜底，不是主要对齐手段。

```python
def align_video_to_audio(video_path: Path, target_duration: float) -> Path:
    """兜底微调：仅处理 < 5% 的时长偏差。"""
    actual = get_video_duration(video_path)
    ratio = actual / target_duration
    if 0.95 <= ratio <= 1.05:
        return ffmpeg_speed_adjust(video_path, target_duration)
    elif ratio < 0.95:
        return ffmpeg_pad_last_frame(video_path, target_duration)
    else:
        # 偏差 > 5% 说明 worker 没有正确消费 timing_manifest，应该在 review 阶段被拦住
        log_warning(f"segment {video_path} duration mismatch > 5%, should have been caught by reviewer")
        return ffmpeg_trim_and_pad(video_path, target_duration)
```

#### 3.2.8 降级策略

- 主方案：`npx hyperframes render` + `ffmpeg`
- 降级方案：如果 hyperframes-cli 不可用，跳过 HTML 片段并在 asset.manifest 中标记 `html_not_rendered: "hyperframes_unavailable"`
- 不降级到 Playwright 截屏——那是在仓库内重建渲染能力，违反边界约束

**规则**：
- 项目存在性 = `configs/{project_id}.json` 存在。`/api/projects` 列表扫描 `configs/` 目录，不扫描 `runtime/`
- 项目运行状态 = `runtime/projects/{project_id}/` 存在且有 events.jsonl
- 刚创建但未执行的项目：在 `configs/` 中有 manifest，在 `runtime/` 中无记录——前端正常显示，状态为"未执行"
- `runtime/` 中残留但 `configs/` 中已删除的项目：不出现在列表中（可提供"清理孤儿运行记录"功能）

#### 3.4.3 运行时配置：project-scoped，非全局

**问题**：之前设计的全局 `/api/config` 写 `.env` 是隐式全局状态，违反 AGENTS.md 约束。

**修正方案**：运行时配置是 **project 级别**，落盘在 `runtime/projects/{project_id}/config.json`。

```typescript
// 项目级运行时配置
interface ProjectRuntimeConfig {
  // 模型配置
  primary_model: string;
  review_model: string;
  worker_model: string;
  reasoning_effort: 'low' | 'medium' | 'high';

  // 执行配置
  dispatch_max_workers: number;
  llm_timeout_seconds: number;

  // TTS 配置
  tts_provider: 'powershell_sapi' | 'f5_tts' | 'command' | 'noop';
  f5_device: 'cpu' | 'cuda';
  f5_model: string;
  f5_reference_audio: string;

  // 调试开关
  progress_log: boolean;
  disable_response_storage: boolean;
}
```

**生命周期定义**：

| 属性 | 值 |
|------|-----|
| 写入者 | 用户通过前端 `/live/projects/:id/settings` 页面 |
| 消费者 | `executor.py`（run_to_review）、`RoleExecutor`（单角色执行）、`prompt-preview` API——三者共用同一套加载逻辑 |
| 落盘路径 | `runtime/projects/{project_id}/config.json` |
| 失效规则 | 项目删除时一并删除；每次执行时读取最新值（不缓存） |
| 优先级 | project config > 环境变量 > 代码默认值 |

**关键约束**：`RoleExecutor` 和调试 API 必须复用同一套项目配置加载逻辑。否则前端看到的 prompt-preview、单角色执行结果和整链路执行结果会跑在不同模型/超时/并发设置上，调试价值打折。

后端改动：
- `GET /api/projects/:id/config` — 读取项目配置（不存在则返回默认值）
- `PUT /api/projects/:id/config` — 写入项目配置
- `load_llm_runtime_config()` 增加 `project_id` 参数，自动查找并加载 `runtime/projects/{project_id}/config.json`
- `RoleExecutor.__init__` 接收 `LLMRuntimeConfig`（由调用方通过 `load_llm_runtime_config(project_id)` 获取）
- `prompt-preview` API 内部也调用 `load_llm_runtime_config(project_id)` 确保预览结果与实际执行一致

#### 3.4.4 Prompt 调试：prompt-preview（非 context-pack）

**问题**：`context-pack` 只渲染默认上下文段（`build_default_prompt_sections`），不包含 recipe sections。真正发给 LLM 的完整 prompt = context sections + recipe sections + user prompt。直接复用 `context-pack` 名字会让调试做偏。

**修正方案**：新增独立的 `prompt-preview` 语义，返回完整的系统提示词。

```python
# 后端新增 API
@router.post("/projects/{project_id}/prompt-preview")
def preview_full_prompt(request: PromptPreviewRequest) -> dict:
    """返回某角色在某阶段的完整系统提示词（context + recipe + shared prompts）。

    这是调试用接口，返回的就是实际会发给 LLM 的 system prompt。
    """
    plan = build_plan_from_manifest(project_id)
    load_execution_task_snapshot(plan)

    bundle = build_prompt_bundle(
        plan=plan,
        session_id=request.session_id,  # 必须传 session_id 才能注入人工反馈
        role_id=request.role_id,
        stage=request.stage,
        recipe=resolve_recipe(request.role_id),
        payload=request.payload or {},
        allow_disallowed_stage=True,
    )
    return {
        "system_prompt": bundle.system_prompt,      # 完整系统提示词
        "user_prompt": bundle.user_prompt,          # 用户提示词
        "prompt_sections": bundle.prompt_sections,  # 分段列表（便于调试哪段来自哪里）
        "context_packet": bundle.packet,            # 原始 context packet
    }
```

关键区别：
- `context-pack`：只有 context_assembly 产出的上下文段，用于持久化和审计
- `prompt-preview`：完整的 system_prompt + user_prompt，包含 recipe sections、shared prompts、role prompts，且需要 session_id 才能注入人工打回反馈

前端调试面板调用 `prompt-preview` 而非 `context-pack`。

### 阶段 3.6：反思与修复机制

**目标**：worker 失败后能生成结构化修复建议，reviewer 打回后能精确定位返工角色。

#### 3.6.1 任务级反思

```python
@dataclass
class FailureReport:
    task_id: str
    failure_type: str  # manim_compile_error | html_validation_error | ...
    root_cause: str
    repair_action: str
    retry_allowed: bool
    prompt_patch: str  # 下次重试时注入的额外约束
```

#### 3.6.2 审核级反思

reviewer 打回时，输出中的 `return_recommendation_if_needed` 字段直接指导返工：
- `target_roles`：退给哪些角色
- `must_fix`：必须修复什么
- `prompt_patch`：下次重试时注入什么额外约束

## 优先级排序

```
Prompt 实战强化 (3.1) > 角色独立初始化 (3.3) > TTS前移+音画同步+HTML接入 (3.2) > 前端控制面板 (3.4) > 权限硬化+反思 (3.5+3.6)
```

理由：
- 3.1 是一切的基础——prompt 不改，脚本质量不会提升，后续所有环节都在执行垃圾输入
- 3.3 是可调试性的基础——角色不能独立测试，就无法快速迭代 prompt 质量
- 3.2 是成片完整性的关键——没有 TTS 前移和音画同步，视频要么缺 HTML 段，要么音画脱节
- 3.4 是可用性的关键——没有前端控制面板，每次调参都要改环境变量重启后端，效率极低
- 3.5+3.6 是稳定性保障——在前四项稳定后再加固
