# ManiMind 上下文与状态设计

## 一、目标

在当前实现中，上下文不仅分长期/短期，还要明确写入者、消费者、生命周期和失效规则，避免隐式全局状态。

## 二、上下文分层

长期上下文目录：

```text
runtime/projects/<project_id>/
```

短期上下文目录：

```text
runtime/sessions/<session_id>/
```

### 静态/动态双态载体

- 静态上下文：已经确认可复用的稳定事实，如 `research.summary`、`glossary`、`storyboard.master`、`review.report`。
- 动态上下文：围绕角色、阶段、会话临时生成或追加的运行时材料，如 `context packet`、失败说明、重试痕迹、任务更新结果。
- 提升规则：子 Agent 先写入动态材料，再由 leader 汇总、提升或提交为项目级静态事实。

补充约束：

- `review.report` 只保存人工确认后的正式审核结论。
- AI reviewer 草案、人工打回意见和返工提示，优先作为会话级动态材料存在，不直接覆盖正式 `review.report`。

更完整的阶段化设计见 [阶段1计划.md](./阶段1计划.md)。

路径蓝图由 `bootstrap.build_runtime_layout(...)` 统一生成。

## 三、上下文记录模型

当前 `ContextRecord` 字段：

- `key`
- `scope`（`long_term` / `short_term`）
- `summary`
- `writer_role`
- `consumer_roles`
- `lifecycle`
- `invalidation_rule`
- `source_ids`
- `sticky`

### 字段约束

- `writer_role`：唯一主写入者，用于避免越权覆盖。
- `consumer_roles`：允许消费该上下文的角色列表。
- `lifecycle`：
  - `project`：跨会话保留
  - `session`：仅当前会话有效
- `invalidation_rule`：触发重写或失效的条件（如 `new_ingest_round`）。

## 四、默认上下文蓝图

由 `workflow.build_context_blueprint(project_id)` 生成，核心项包括：

- `research.summary`
- `glossary`
- `formula.catalog`
- `style.guide`
- `narration.script`
- `storyboard.master`
- `asset.manifest`
- `review.report`
- `session.handoff`

补充说明：

- `review.report`：正式、已人工确认的审核结论。
- 人工打回意见不建议直接放入默认长期蓝图，而应按会话轮次进入短期动态上下文。

## 五、上下文装配机制

由 `context_assembly.build_context_packet(...)` 负责：

1. 根据角色 `AgentMode` 选择默认上下文集合。
2. 合并角色 `required_inputs`。
3. 生成带治理字段的 `context_specs`。
4. 注入统一约束：
   - 长期上下文只写 `runtime/projects/...`
   - 短期上下文只写 `runtime/sessions/...`
   - 审核未通过不得进入后处理
5. 输出 `write_targets` 和 `output_contract`。

### 人工打回的上下文植入位置

当人工 reviewer 选择 `return` 时，建议增加两类会话级动态上下文：

1. `review.return.memo`
   - 内容：人工打回意见原文、影响任务、影响产物、优先级、是否允许局部重试
   - 建议写入者：`human_reviewer`
   - 建议消费者：`lead`、`coordinator`、`reviewer`、目标 worker
2. `review.return.prompt_patch`
   - 内容：下一轮执行时需要直接注入的返工指令
   - 形式：短文本或结构化片段，聚焦“必须改什么、保持什么、通过条件是什么”

建议物理位置：

```text
runtime/sessions/<session_id>/review-returns/
```

建议记录字段：

- `session_id`
- `review_round`
- `target_role`
- `target_task_ids`
- `target_output_keys`
- `reason`
- `must_fix`
- `should_keep`
- `prompt_patch`
- `created_at`

## 六、提示词分段与缓存

`context_assembly` 中实现了：

- `PromptSection`
- `PromptSectionCache`
- `section(...)`
- `volatile_section(...)`

用途：

- 把系统提示词拆成稳定段（可缓存）和波动段（每轮重算）
- 减少重复拼接成本，保证角色约束注入一致

### 人工打回的 prompt 植入位置

人工打回意见不应混入普通 `summary`，而应作为独立高优先级段注入。

推荐顺序：

```text
role
human_review_feedback
context
output
guardrails
```

也就是：

1. 先告诉目标角色“你是谁、当前在哪个阶段”
2. 紧接着注入“这轮人工为什么打回、必须怎么改”
3. 再给常规上下文与输出边界

这样可以避免人工返工要求被淹没在普通上下文列表里。

### 人工打回的目标角色分发规则

- 如果问题属于叙事、分镜、章节组织，优先注入 `coordinator`
- 如果问题属于具体媒介片段质量，优先注入对应 `html_worker` / `manim_worker` / `svg_worker`
- 如果问题属于全局数学事实、术语或风格规则，同时注入 `coordinator` 与受影响 worker
- `lead` 永远可见所有人工打回记录，用于决定是否重新派发或升级阻塞

## 七、任务状态与上下文联动

任务状态由 `ExecutionTask` + `task_board` 管理：

- 只有任务 owner（或 `lead`）可改状态
- blocker 未完成时禁止推进
- `review.outputs` 是强制验证关卡
- 在“前置非审核任务全部完成但审核未完成”时触发 verification nudge
- `blocked` 通过依赖和事件原因表达，不额外增加任务状态枚举
- 阶段 1 约定：收到 `worker.blocker` 时，任务可维持 `in_progress`，但查询快照应额外提供 `blocked_reason` / `blocked_at`

这保证了上下文写入顺序与阶段推进顺序一致。

## 八、当前缺口与下一步

### 1. 人工审核上下文还没有落成固定载体

当前仓库还缺少对以下信息的正式落盘：

- AI reviewer 草案
- 人工审核通过
- 人工审核打回
- 打回后的 prompt patch

这意味着虽然可以表达 `review.outputs` 是关卡，但还没有把“人工确认/人工打回”的返工闭环真正编排化。

### 2. `consumer_roles` 需要落成硬过滤

当前上下文治理字段已经存在，但 `context-pack` 还需要按 `consumer_roles` 严格过滤上下文，避免默认把 long-term sticky 内容广播给不应读取的角色。

### 3. `stage_allowed` 需要从提示变成门禁

当前 `context-pack` 可返回 `stage_allowed`，但默认仍允许继续工作。长期目标应改为：

- 默认阻断非法阶段请求
- 仅在显式允许时跳过门禁

### 4. 真正的 assembled prompt 要放在状态闭环之后

`build_context_packet(...)` 当前更接近 metadata packet。只有当 runtime 成为唯一事实源后，才适合继续推进：

- 真实上下文正文读取
- prompt section 渲染预览
- assembled prompt 输出
