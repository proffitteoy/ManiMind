## 优先修复：已知 Bug（P0）

### Bug-1：`role_executor.py` 调用签名错误

**现象**：`generate_text_for_role` 和 `generate_json_for_role` 在 `llm_client.py` 中定义为 keyword-only 参数（`*, cfg, role_id, instructions, prompt`），但 `role_executor.py:127-140` 使用 positional 方式传参。独立执行 `RoleExecutor.execute()` 时会 TypeError。

**修复方案**：
```python
# role_executor.py:127-131 当前写法（正确，已经是 keyword 传参）
text, meta = generate_text_for_role(
    self._cfg,        # ← 这里是 positional！
    self.role_id,     # ← 这里也是 positional！
    instructions=bundle.system_prompt,
    prompt=bundle.user_prompt,
)
```

实际检查发现 `role_executor.py` 第 127-140 行的调用方式是：
```python
generate_text_for_role(
    self._cfg,
    self.role_id,
    instructions=...,
    prompt=...,
)
```

而 `llm_client.py:462` 和 `479` 的签名是：
```python
def generate_json_for_role(*, cfg, role_id, instructions, prompt): ...
def generate_text_for_role(*, cfg, role_id, instructions, prompt): ...
```

`*` 表示所有参数都是 keyword-only。`self._cfg` 和 `self.role_id` 作为 positional 传入会直接 TypeError。

**修复**：改为显式 keyword 传参：
```python
text, meta = generate_text_for_role(
    cfg=self._cfg,
    role_id=self.role_id,
    instructions=bundle.system_prompt,
    prompt=bundle.user_prompt,
)
parsed, meta = generate_json_for_role(
    cfg=self._cfg,
    role_id=self.role_id,
    instructions=bundle.system_prompt,
    prompt=bundle.user_prompt,
)
```

**验证**：单独实例化 `RoleExecutor("explorer")` 并调用 `execute()`，确认不再 TypeError。

### Bug-2：后处理拼接逻辑的"媒介择一"问题

**现象**：`post_produce.py:_collect_segment_videos()` 对每个 segment 使用 `continue` 跳出，导致一旦找到 Manim 产物就跳过 HTML 产物（反之亦然）。最终成片只拼接单一媒介类型的视频，无法实现"HTML 负责引子/桥接、Manim 负责公式核心"的联合成片。

**根因**：第 153-157 行的 `continue` 逻辑：
```python
# 找到 manim mp4 后直接 continue，跳过了同 segment 的 html 产物
for item in files:
    if ... item.lower().endswith(".mp4") ...:
        ordered_videos.append(item)
        break
    else:
        continue
    continue  # ← 跳过 html_key 检查
```

**修复方案**：改为按分镜顺序联合收集所有媒介产物。对 HYBRID segment，按 `manim → html → svg` 优先级收集，但不互斥跳过。具体策略：

1. 对每个 segment，收集该 segment 下所有已 approved 的 mp4（可能来自 manim、html、svg）
2. 按 `storyboard_outline` 中的 `scene_beats` 或 `worker_instructions` 决定拼接顺序
3. 如果 segment 只有一种媒介产物，直接使用
4. 如果有多种，按 planner 指定的 `primary_worker_path` 决定主产物，其余作为补充片段

**简化版修复**（最小改动）：去掉外层 `continue`，让每个 segment 可以同时收集 manim 和 html 的 mp4：
```python
for segment in plan.segments:
    segment_videos: list[str] = []
    
    manim_key = f"{plan.project_id}.manim.{segment.id}.approved"
    record = record_by_key.get(manim_key)
    if record:
        # 收集 manim mp4
        ...
    
    html_key = f"{plan.project_id}.html.{segment.id}.approved"
    record = record_by_key.get(html_key)
    if record:
        # 收集 html mp4
        ...
    
    svg_key = f"{plan.project_id}.svg.{segment.id}.approved"
    record = record_by_key.get(svg_key)
    if record:
        # 收集 svg mp4
        ...
    
    ordered_videos.extend(segment_videos)
```

**验证**：构造一个 HYBRID segment 同时有 manim 和 html 产物的测试用例，确认最终 concat-list.txt 包含两者。

## 架构改造：Stage Runner 拆分（P1）

### 目标

将 `run_to_review` 从 1300 行的中心大函数拆分为可独立运行的 stage runner，支持：
- 单阶段重跑（`rerun plan`）
- 单 segment 重跑（`rerun html_worker for segment_003`）
- 断点续跑（从上次失败的 stage 继续）

### 阶段语义说明

Stage Runner 不引入新的 `PipelineStage` 枚举值。Runner 的粒度是 `run_to_review` 内部的逻辑步骤，不是 pipeline 状态机的节点。具体映射：

| Runner | 对应 PipelineStage | 说明 |
|--------|-------------------|------|
| IngestRunner | INGEST | 归档输入 |
| SummarizeRunner | SUMMARIZE | 包含 explorer + lead 两步，explorer 不独立暴露为 stage |
| PlanRunner | PLAN | 包含 planner + coordinator 两步 |
| DispatchRunner | DISPATCH | 并发 worker 调度 |
| ReviewRunner | REVIEW | reviewer draft 生成 |
| PostProduceRunner | POST_PRODUCE | 后处理拼接 |
| PackageRunner | PACKAGE | 打包交付 |

CLI 的 `rerun` 命令对外暴露的是 runner 名称（如 `rerun summarize`），不是 PipelineStage 枚举。`rerun summarize` 会重跑 explorer + lead 两步。不存在 `rerun explore` 这个独立命令。

### 设计

```text
src/manimind/
├─ stages/
│  ├─ __init__.py          # StageRunner 协议定义
│  ├─ ingest.py            # IngestRunner
│  ├─ summarize.py         # SummarizeRunner (含 explorer + lead)
│  ├─ plan.py              # PlanRunner (planner + coordinator)
│  ├─ dispatch.py          # DispatchRunner (并发 worker 调度)
│  ├─ review.py            # ReviewRunner
│  ├─ post_produce.py      # PostProduceRunner
│  └─ package.py           # PackageRunner
├─ stage_orchestrator.py   # 替代 run_to_review 的编排入口
└─ executor.py             # 保留为兼容入口，内部委托 stage_orchestrator
```

### StageRunner 协议

```python
from dataclasses import dataclass, field
from typing import Any, Protocol

@dataclass
class StageInput:
    plan: ProjectPlan
    session_id: str
    cfg: LLMRuntimeConfig
    prompt_cache: PromptSectionCache

@dataclass
class StageOutput:
    stage: PipelineStage
    success: bool
    error: FailureCategory | None = None
```

说明：
- **去掉 `upstream_outputs` 和 `outputs` 字段**。Runner 之间不通过内存态传递数据，而是通过已有的 `write_output_key` / `has_output_key` 机制读写持久化产物。这与仓库"不允许用隐式全局状态驱动多 Agent 协作"的红线一致。
- **去掉 `can_resume` 方法**。断点续跑通过读取 `execution-tasks.json` 中的任务状态判断：如果某个 runner 对应的 task 已经 COMPLETED，orchestrator 跳过该 runner。这复用了 `runtime_store.py` 已有的 `load_execution_task_snapshot` 机制，不引入新的持久化层。

### 断点续跑实现

不新增"stage snapshot"持久化层。利用现有机制：

1. `runtime/projects/<project_id>/execution-tasks.json` 记录每个 task 的状态
2. `output_path_for_key` 负责 output_key 的真实落盘：
   - `*.session.*` → `runtime/sessions/<session_id>/artifacts/`
   - `*.approved` / `*.asset.manifest` → `outputs/<project_id>/artifacts/`
   - 其他 key → `runtime/projects/<project_id>/artifacts/`
3. orchestrator 启动时调用 `load_execution_task_snapshot(plan)` 回填状态
4. 对每个 runner，检查其对应 task 是否已 COMPLETED：
   - 已完成 → 跳过
   - IN_PROGRESS 或 PENDING → 执行
5. runner 内部通过 `has_output_key` 判断上游产物是否存在，不依赖内存传递

这样断点续跑的状态完全由已有的 task snapshot + artifact 文件决定，不引入新的生命周期概念。

### 迁移策略

1. 先把 `run_to_review` 中每个阶段的逻辑提取为独立函数（不改变行为）
2. 为每个函数包装 StageRunner 接口
3. 新增 `stage_orchestrator.py` 按 DAG 顺序调用 runners
4. `run_to_review` 改为调用 `stage_orchestrator.run_all()`
5. CLI 新增 `rerun <runner_name>` 命令
6. 同步更新 `docs/architecture.canvas` 和 `docs/通用项目架构模板.md`

### 实施顺序

| 步骤 | 内容 | 预计工作量 |
|------|------|-----------|
| 1 | 提取 ingest/summarize 为独立函数 | 小 |
| 2 | 提取 plan (planner+coordinator) 为独立函数 | 中 |
| 3 | 提取 dispatch (worker 调度) 为独立函数 | 中 |
| 4 | 提取 review 为独立函数 | 小 |
| 5 | 实现 StageRunner 协议和 orchestrator | 中 |
| 6 | 实现 `rerun` CLI 命令 | 小 |
| 7 | 同步更新架构文档 | 小 |

## 可观测性工程（P2）

### 目标

让每次模型调用可回答：模型当时看到了什么、输出了什么、为什么进入下一步、失败发生在哪一层。

### 设计

在现有 `events.jsonl` 和 `context-packets` 基础上，新增结构化 trace：

```python
@dataclass
class LLMTrace:
    trace_id: str
    role_id: str
    stage: str
    task_id: str
    timestamp: str
    # 输入
    input_context_keys: list[str]
    prompt_sections: list[str]
    model_route: str
    # 输出
    model_output_excerpt: str  # 前 2000 字符
    parsed_output_keys: list[str]
    schema_validation: str  # "pass" | "fail:reason"
    # 产物
    artifact_files: list[str]
    render_command: str | None
    render_exit_code: int | None
    # 性能
    duration_ms: int
    token_usage: dict[str, int] | None
    retry_count: int
    failure_reason: str | None
```

### 落盘路径

Trace 包含 `prompt_sections`、`model_output_excerpt`、`retry_count`、`failure_reason` 等会话级动态数据，属于短期协作材料。按仓库约束（AGENTS.md、强前置条件约束.md），写入路径为：

```
runtime/sessions/<session_id>/traces/<timestamp>-<role_id>-<stage>.json
```

不写入 `runtime/projects/<project_id>/`。项目级目录只保留聚合索引（trace 数量、失败率等统计摘要），不存原始 trace。

### 生命周期契约

| 属性 | 值 |
|------|-----|
| 写入者 | `_call_json_role` / `generate_json_for_role` / `generate_text_for_role` 调用层 |
| 消费者 | CLI `trace` 命令、前端 trace 面板、failure taxonomy 分类器 |
| 失效规则 | 随 session 生命周期，session 归档时可清理 |
| 持久化位置 | `runtime/sessions/<session_id>/traces/` |

### 实施

1. 在 `_call_json_role`、`generate_json_for_role`、`generate_text_for_role` 调用前后记录 trace
2. Trace 写入 `runtime/sessions/<session_id>/traces/` 目录
3. 项目级 `runtime/projects/<project_id>/trace-summary.json` 只存聚合统计
4. CLI 新增 `trace <project_id> --session <session_id> [--stage] [--role] [--failed-only]` 查询命令
5. 前端 `/live` 页面新增 trace 面板

## 失败分类工程（P2）

### Failure Taxonomy

```python
class FailureCategory(str, Enum):
    INPUT_MISSING = "input_missing"
    PDF_EXTRACT_FAILED = "pdf_extract_failed"
    LLM_JSON_INVALID = "llm_json_invalid"
    SCHEMA_VALIDATION_FAILED = "schema_validation_failed"
    WORKER_OUTPUT_INVALID = "worker_output_invalid"
    HTML_RENDER_FAILED = "html_render_failed"
    MANIM_LATEX_ERROR = "manim_latex_error"
    MANIM_SYNTAX_ERROR = "manim_syntax_error"
    FFMPEG_MERGE_FAILED = "ffmpeg_merge_failed"
    AUDIO_VIDEO_DURATION_MISMATCH = "audio_video_duration_mismatch"
    REVIEW_NOT_COMPLETED = "review_not_completed"
    MISSING_REQUIRED_OUTPUTS = "missing_required_outputs"
    TTS_SYNTHESIS_FAILED = "tts_synthesis_failed"
    TIMEOUT = "timeout"
    UNKNOWN = "unknown"
```

### 与 Stage Runner 集成

每个 StageRunner 的 `run()` 方法在失败时返回 `StageOutput(success=False, error=FailureCategory.XXX)`，orchestrator 根据 category 决定：
- `LLM_JSON_INVALID` → 自动 retry with contract_hint
- `MANIM_LATEX_ERROR` → 触发 manim repair 流程
- `HTML_RENDER_FAILED` → 降级为静态截图
- `REVIEW_NOT_COMPLETED` → 阻塞等待人工
- `TIMEOUT` → 记录并允许手动 rerun

## Schema Contract 工程（P2）

### 目标

prompt 要求什么字段 → normalize 保留什么字段 → reviewer 检查什么字段 → 前端展示什么字段。全链路一致。

### 实施

1. 定义 `contracts/` 目录，每个角色输出一个 JSON Schema 文件
2. normalize 函数改为 schema-driven：读取 schema，保留所有 schema 中定义的字段
3. reviewer 的 `evidence_checks` 基于 schema 自动生成检查项
4. 前端根据 schema 动态渲染字段

```text
src/manimind/contracts/
├─ planner_output.json
├─ coordinator_output.json
├─ html_worker_output.json
├─ manim_worker_output.json
├─ svg_worker_output.json
└─ reviewer_output.json
```

## 角色边界工程（P3）

### 目标

每个角色被 harness 限定在可审计的工作域里，不能越权。

### 实施

现有 `AgentProfile` 已定义 `allowed_stages` 和 `output_contract`，但运行时未强制执行。需要：

1. 在 `build_prompt_bundle` 中增加 stage 校验（已有 `allow_disallowed_stage` 参数，默认应为 False）
2. **ownership 校验放在 orchestrator / role executor 层**，不下沉到 `write_output_key`。原因：`artifact_store.write_output_key(plan, session_id, key, payload)` 的签名不携带 `role_id`，且被人工审核流程（`review_workflow.py`）和系统流程直接调用。如果硬塞 ownership 校验，要么改穿所有调用栈，要么退化成隐式全局状态，两条路都违反架构约束。正确做法是在 `RoleExecutor.execute()` 和 `stage_orchestrator` 调度层做前置校验：执行前检查当前 role 是否拥有即将写入的 output_key。
3. 在 `_call_json_role` 返回后增加 output schema 校验
