# 2026-05-11 真实多角色 API 与 F5 后处理改造

## 目标

本次改造的目标有四个：

1. 删除模型回退逻辑，改成显式的主路由 / 审核路由 / worker 路由。
2. 把 planner、coordinator、reviewer、html/svg/manim worker 都接到真实 API，而不是本地模板或硬编码流程。
3. 把 prompt 从执行器中拆出来，形成可复用、可动态拼装、但保留基础角色配方的独立系统。
4. 把字幕、配音、最终视频打包收敛到 `finalize` 主流程，并接入项目内固定参考音频的 F5-TTS。

## 本次改动

### 1. 多路由 LLM 配置收敛

文件：`src/manimind/llm_client.py`

- 使用 `primary / review / worker` 三条显式路由。
- `lead` 默认走 `primary`。
- `reviewer` 走 `review`。
- `explorer / planner / coordinator / html_worker / svg_worker / manim_worker` 走 `worker`。
- 统一使用 Responses API。
- 删除原有 fast model / chat completions / provider 兼容回退链。

当前建议的环境变量：

- `OPENAI_API_KEY`
- `MANIMIND_MODEL_BASE_URL`
- `MANIMIND_MODEL`
- `MANIMIND_REVIEW_MODEL`
- `MANIMIND_WORKER_MODEL`
- `MANIMIND_WORKER_API_KEY`
- `MANIMIND_MODEL_REASONING_EFFORT`
- `MANIMIND_REVIEW_MODEL_REASONING_EFFORT`
- `MANIMIND_WORKER_MODEL_REASONING_EFFORT`

说明：

- 代码中不写入任何 API key。
- worker 路由默认沿用主路由 base URL，仅单独切换模型名和 key。

### 2. 独立 prompt 系统

文件：`src/manimind/prompt_system.py`

新增独立提示词系统，核心点：

- `PromptRecipe`：定义角色的 focus / deliverable / task_brief / response_contract / extra_rules。
- `build_prompt_bundle(...)`：从 context packet、默认 prompt sections 和角色 recipe 动态拼装完整系统提示词。
- 角色配方独立维护：
  - `explorer_recipe()`
  - `lead_summary_recipe()`
  - `planner_recipe()`
  - `coordinator_recipe()`
  - `reviewer_recipe()`
  - `html_worker_recipe()`
  - `svg_worker_recipe()`
  - `manim_generate_recipe()`
  - `manim_repair_recipe()`

这样做的目的：

- 避免在 `executor.py` 里把角色能力写死成散乱字符串。
- 允许后续继续扩展 prompt 片段，而不需要重写执行器。
- 保留每个角色最初的基础配方，避免动态拼装后失去稳定的底座。

### 3. 执行器改成真实多角色链路

文件：`src/manimind/executor.py`

当前 `run_to_review(...)` 过程为：

1. ingest 读取源文档。
2. explorer 提取结构化发现。
3. lead 沉淀 `research.summary / glossary / formula.catalog / style.guide`。
4. planner 产出结构化镜头规划。
5. coordinator 产出脚本、storyboard 和 worker handoff。
6. 根据 planner 的 `primary_worker_path` 决定当前分镜实际执行哪条 worker 路由；未被选中的候选任务会被显式标记为 `skipped_by_plan`。
7. html/svg/manim worker 对被选中的分镜调用真实模型生成产物。
8. reviewer 只输出审核草案，不直接放行。

额外修复：

- `style.guide` 现在会真正落盘，不再只存在于内存里。
- glossary 提取不再硬编码某些数学术语白名单，而是从源文档中抽取候选项。

### 4. worker 侧不再走假产物

文件：`src/manimind/worker_adapters.py`

- HTML worker：真实生成完整 HTML 文档。
- SVG worker：真实生成完整 SVG 文档。
- Manim worker：真实生成 Python 场景代码并尝试渲染。
- Manim 渲染失败时，进入 repair loop，用同一路由继续修复。
- 不再写出“fallback note”伪产物来冒充成功。

### 5. 后处理并入 `finalize`

文件：`src/manimind/post_produce.py`

`finalize_delivery(...)` 现在负责：

- 收集 approved records。
- 读取 narration script。
- 生成 `outputs/<project_id>/subtitles/narration.srt`。
- 调用 TTS 生成音频。
- 合并 Manim 片段为最终视频。
- 生成 `asset.manifest`。

当前视频合并策略：

- 优先按 segment 顺序收集 Manim approved mp4。
- 若没有按段的 Manim mp4，才退回到任意 approved mp4。

这也是首次运行中“更像公式展示视频”的一个直接原因：当前最终视频拼接仍以 Manim 片段为主，而不是按完整叙事骨架收集 HTML 引子、桥接段和公式核心段。

### 6. F5-TTS 正式接入 TTS provider

文件：

- `src/manimind/tts.py`
- `scripts/finalize-with-f5-tts.ps1`
- `scripts/f5_tts_generate.py`

新增 `f5_tts` provider：

- `finalize --tts-provider f5_tts` 可以直接走项目内固定参考音频。
- 默认参考音频路径：
  `runtime/projects/<project_id>/voice/selena_reference.m4a`
- 默认参考文本路径：
  `runtime/projects/<project_id>/voice/selena_reference.txt`
- 默认缓存路径：
  `runtime/projects/<project_id>/voice/hf-cache/`

`scripts/finalize-with-f5-tts.ps1` 现在不再通过 `MANIMIND_TTS_COMMAND` 间接调用，而是设置 F5 专用环境变量后直接调用：

```powershell
python -m manimind.main finalize <manifest> --tts-provider f5_tts
```

## 真实案例清单

新增：`configs/cauchy-backward-induction-real.json`

对应源文档：`tests/柯西的反向归纳法.md`

设计原则：

- 共 4 个主段，对应原文 4 个核心主题。
- 并非所有分镜都设为 `hybrid`；背景型分镜优先走 HTML，数学推导型分镜保留 Manim 候选。
- planner 通过 `segment_priorities[*].primary_worker_path` 决定当前轮实际执行的 worker。
- 执行器会把未被选中的候选任务记为 `skipped_by_plan`，而不是强行全跑。

## 首次运行发现与下一轮改造要求

以下三项来自首次真实运行观察，属于下一轮必须进入改造清单的结构性问题，而不是单点体验 bug。

### 1. 配音过快，且时序位置放错了

现象：

- 当前配音速度过快，几乎无法听清。
- 当前主流程是先生成视觉片段，再在 `finalize` 阶段统一做 TTS。
- 这导致画面时长不是由真实旁白时长驱动，而是由预估秒数或渲染产物反推。

判定：

- 当前流程更像“先做画面，再塞音频”，不适合讲解型数学视频。
- 对于科普讲解视频，音频应先成为时长基准，再驱动画面节拍与每段镜头长度。

下一轮要求：

1. 在拿到 `narration.script` 之后，先执行配音，再进入画面制作。
2. 需要新增“音频对齐”步骤，至少产出：
   - 每个 part 的旁白音频
   - 每个 part 的真实时长
   - 可供 worker 消费的 timing manifest
3. 后续 HTML / Manim / SVG 的每一段时长，默认以对应配音时长为主，而不是靠 `estimated_seconds` 硬估。
4. `tts.py` 需要支持可控语速；无论使用 F5-TTS 还是其他 provider，都必须把语速纳入配置与审核项。

建议目标链路：

```text
ingest -> summarize -> plan.storyboard -> synthesize.narration -> align.segment_timing -> render visuals -> review -> package
```

受影响模块：

- `src/manimind/executor.py`
- `src/manimind/post_produce.py`
- `src/manimind/tts.py`
- `src/manimind/workflow.py`

### 2. 媒介分发不应让所有片段都跑 html/svg/manim

现象：

- 首次运行中观察到所有片段都在尝试跑 `svg/html/manim`。
- 这和期望不一致。你真正需要的是“按分镜语义路由”，而不是“所有媒介默认并发候选”。

判定：

- 公式推导、公式可视化、数学结构变化，优先应走 `manim_worker`。
- 开头引入、段间衔接、弱科普说明、轻量信息卡和氛围铺垫，优先应走 `html_worker`。
- `svg_worker` 应是补充型能力，而不是所有片段默认跑一遍的主路径。

下一轮要求：

1. planner 必须对每个 segment 显式判断内容类型，而不是只给泛化的 `hybrid`。
2. coordinator 必须把分镜分成至少三类：
   - `intro / bridge / weak_explainer` -> HTML
   - `formula_core / derivation / proof_visualization` -> Manim
   - `diagram_overlay / optional_relation_graph` -> SVG（仅在明确需要时）
3. `workflow.build_worker_tasks(...)` 的默认派发策略需要进一步收敛，避免因为 `hybrid` 或公式字段存在而把所有媒介都挂上去。
4. reviewer 需要新增一条检查：媒介是否与 segment 目标匹配，而不是只看产物是否存在。

受影响模块：

- `src/manimind/workflow.py`
- `src/manimind/executor.py`
- `src/manimind/prompt_system.py`
- `src/manimind/worker_adapters.py`

### 3. 当前成片只有公式可视化，缺少引子和公式之间的衔接

现象：

- 当前视频更像“公式展示集锦”。
- 缺少开头的引子。
- 缺少公式与公式之间的过渡、解释桥段和弱科普连接段。

判定：

- 这不是后处理问题，而是 storyboard coverage 不完整。
- 现在的分镜生成还没有把“引入 - 解释 - 推导 - 过渡 - 收束”作为强制叙事骨架。

下一轮要求：

1. planner 和 coordinator 生成分镜时，必须至少覆盖：
   - 开头引子
   - 关键公式段
   - 公式之间的桥接段
   - 结尾收束或总结段
2. HTML worker 要承担“引子 + 过渡 + 弱科普”主责任，而不是只做可有可无的补充动画。
3. reviewer 要把“是否形成完整叙事链，而不是公式堆叠”作为硬性审核项。
4. `storyboard_master` 建议补充 segment 语义标签，例如：
   - `intro`
   - `bridge`
   - `formula_core`
   - `summary`

受影响模块：

- `src/manimind/prompt_system.py`
- `src/manimind/executor.py`
- `src/manimind/review_workflow.py`
- `src/manimind/workflow.py`

### 汇总：下一轮改造优先级

1. 先改流程顺序：脚本后先配音，再按真实时长做画面。
2. 再改媒介路由：Manim 只做公式核心，HTML 承担引子和桥接，SVG 变成可选增强。
3. 最后补审核约束：没有引子、没有桥接、只有公式堆叠的成片，不允许过审。

## 真实运行建议

推荐使用以下环境组合：

- 主路由：`gpt-5.5`
- 审核路由：`gpt-5.5`
- worker 路由：`deepseek-v4-flash`
- 后处理配音：`f5_tts`

推荐执行顺序：

```powershell
python -m manimind.main run-to-review configs/cauchy-backward-induction-real.json --session-id real-cauchy-001
python -m manimind.main human-review configs/cauchy-backward-induction-real.json approve --session-id real-cauchy-001 --reason "real run approved"
python -m manimind.main finalize configs/cauchy-backward-induction-real.json --session-id real-cauchy-001 --tts-provider f5_tts
```

## 未完成与风险

1. 真实联网运行依赖本机 Python、Manim、ffmpeg 和外网访问权限；这一步需要实际执行验证。
2. F5-TTS 依赖额外 Python 包与模型缓存，第一次运行会比后续慢很多。
3. 当前字幕文件已纳入主流程，但仍以单独 `.srt` 形式输出；是否要把字幕进一步压制进最终视频，可在下一轮决定。
