# ManiMind 前端控制台骨架方案

本文档对应新增目录 `frontend/manimind-console/`，用于说明当前前端骨架的目标、结构和后续接线方式。

## 一、目标

本轮不是做完整产品，而是先把 ManiMind 的“编排控制台”界面壳搭起来，验证三件事：

1. 项目状态、阶段流、任务看板和 Runtime 是否能在一个页面里形成闭环感知。
2. 前端代码是否已经按后续 API 接入做好职责拆分。
3. 控制台是否明确遵守“前端不直接读写 runtime 文件，只经 backend API 访问状态”的架构边界。

## 三、当前实现范围

已落地内容：

- 独立前端子目录：`frontend/manimind-console/`
- 可运行的 Next.js 首页骨架
- 控制台首页布局与视觉基调
- 阶段流、任务看板、Runtime、Context Packet、事件日志、Agent 状态、产物预览、MVP 能力清单
- `mock data -> future API` 的明确替换点
- `/live` 已接入 `backend/` 的项目、任务、上下文、审核证据、产物与 trace 接口

暂未落地内容：

- 项目列表页、任务详情页、上下文详情页
- SSE 实时刷新
- 表单提交、任务状态写回、权限与鉴权

职责划分：

- `src/app/`：页面入口与全局视觉基线。
- `src/components/console/`：控制台壳层与首页业务模块。
- `src/components/ui/`：不依赖领域语义的轻量展示组件。
- `src/data/console-demo.ts`：当前首页 mock 数据，未来替换成 API 响应映射层。
- `src/lib/`：纯工具函数。

## 五、页面信息结构

首页按控制台语义组织，不做“聊天窗口式 AI 工具”：

1. 左侧导航
   - 总览、项目、任务、上下文、Runtime、产物、审核、设置
2. 顶部项目区
   - 项目标题、运行状态、操作按钮
3. 阶段流
   - `INGEST -> SUMMARIZE -> PLAN -> DISPATCH -> REVIEW -> POST_PRODUCE -> PACKAGE -> DONE`
4. 执行核心区
   - 任务看板
   - Runtime 面板
   - Context Packet 面板
   - 事件日志
5. 运营概览区
   - 指标卡片
   - Agent 状态
   - 产物预览
   - MVP 能力清单
6. 底部闭环步骤
   - 用于强调“输入 -> 计划 -> 分发 -> 审核 -> 交付”的整体路径

## 六、与现有后端的边界

前端骨架遵守以下边界：

- 前端不直接读 `runtime/projects/*`
- 前端不直接写 `runtime/sessions/*`
- 前端不调用 `src/manimind/` 内部模块
- 前端只通过 `backend/` 暴露的 HTTP API 访问状态

后续推荐替换关系：

| 当前 mock 字段 | 后续接口来源 |
|:---|:---|
| `stages` | `GET /api/projects/{project_id}/runtime` |
| `taskColumns` | `GET /api/projects/{project_id}/tasks` |
| `runtimeAssets` | `GET /api/projects/{project_id}/runtime` |
| `contextItems` | `POST /api/projects/{project_id}/context-pack` |
| `events` | `GET /api/projects/{project_id}/events` |

## 七、为什么先做静态骨架

当前仓库的主矛盾仍然是 Runtime 状态闭环，而不是前端细节。因此先做静态骨架有两个好处：

- 可以先固化信息架构和模块边界，不必等所有 API 完成才开始前端。
- 后端状态结构一旦稳定，只需要替换 `src/data/console-demo.ts` 的来源，而不是重写整套页面。

## 九、下一步建议

建议按以下顺序推进：

1. 补齐 `backend/` 的项目列表与事件流接口。
2. 给前端新增 `lib/api.ts`，把 mock 数据替换成真实查询。
3. 增加项目页与任务页路由，不再把所有信息堆在首页。
4. 接入手动任务推进按钮，对接 `task-update`。
5. 在状态闭环稳定后，再引入 SSE 或轮询刷新。
